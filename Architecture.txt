================================================================================
                     ENROLLMENT & TEACHING ASSIGNMENT MODEL                     
================================================================================


Course existence is independent of both students and teachers.
A course is created once (admin.html -> Create Course) and then:

  TEACHER ASSIGNMENT                     STUDENT ENROLLMENT
  (who teaches it)                       (who's taking it)
        |                                       |
        v                                       v
  course_teachers table                   enrollments table
  (course_id, teacher_id)                 (student_id, course_id)
        |                                       |
        v                                       v
  teacher.html filters                    index.html filters
  "My Courses" to only                    "My Courses" to only
  courses this teacher_id                 courses this student_id
  is assigned to                          is enrolled in

Neither table determines the other -- a teacher being assigned to a
course does NOT enroll any students, and vice versa. They're two
independent many-to-many relationships sharing the same courses table.

Enrollment can be created three ways (admin.html -> Enrollment / Auto-Enroll Rules):

  1. MANUAL          admin picks one student + one course, clicks Enroll.
                      Works for any institution type -- the universal fallback.

  2. AUTO (PREFIX)    University-style. Matches every student whose Portal ID
                      starts with a given prefix (e.g. "DCS") into one course.
                      Relies on registration-number conventions that only
                      make sense where IDs encode a program/department.

  3. AUTO (GRADE/     CBC-style. Matches every student tagged with a given
     STREAM)          grade (and optionally stream) into one course -- because
                      CBC school IDs don't encode anything useful to match on;
                      the "class" itself (e.g. Grade 10 East) is what defines
                      who's in a course, not a naming pattern.

grade/stream are optional fields on the users table -- blank for students
at institutions that don't use them (e.g. university students using the
prefix-match rule instead).

================================================================================
                             CORE DESIGN PRINCIPLES                             
================================================================================


  UI NEVER READS DIRECTLY FROM SUPABASE FOR COURSE DATA
  (student/teacher pages -- admin.html is the one exception,
   since it manages live data and was never meant to work offline)

      WRONG (student/teacher pages):
          UI ------> SUPABASE

      CORRECT (student/teacher pages):
          UI -> INDEXEDDB <-> SUPABASE (syncFromSupabase(), Realtime)

      admin.html (intentional exception):
          UI ------> SUPABASE directly, every action
                     (optimistic update + Realtime row-patch,
                      excluded from Service Worker caching)


  FIREBASE AUTH = IDENTITY ONLY, NOT COURSE DATA

  Firestore still holds exactly one thing per user: their auth profile
  (uid, role, portalId, displayName, email). Everything else -- courses,
  lectures, assignments, enrollments, teaching assignments -- lives in
  Supabase, because Supabase supports relational joins (course <-> teacher
  <-> student) that Firestore doesn't do well.


  REALTIME PATCHES ROWS, NEVER REFETCHES WHOLE TABLES

  Early version of admin.html's Realtime handlers called fetchXyz() (a
  full table refetch) on every single change. This raced against
  optimistic local updates -- clicking Enroll would add a row locally,
  the write would trigger the table's own Realtime echo, and the refetch
  would sometimes land with stale data and erase the new row ("immediately
  disenrolls" bug). Fixed by patching only the row in the event payload
  (payload.new / payload.old) directly into local state -- no network
  round-trip per event.


  ADMIN.HTML IS NEVER SERVICE-WORKER CACHED

  Every other HTML page uses cache-first (works offline, background-
  refreshes). admin.html manages live enrollment/assignment data and was
  explicitly carved out to always hit the network -- caching it caused
  changes to appear delayed by an unpredictable number of hard refreshes.


  NO CUSTOM SERVER FOR APP LOGIC -- SUPABASE + FIREBASE ONLY

  Supabase Postgres + Storage + Realtime, plus Firebase Auth.
  No Node/Express backend, no Cloud Functions.


  NO ENCODING -- RAW MP4 + RANGE REQUESTS


  NO USER ACTION TO CACHE -- AUTO ON CONNECTIVITY

This ensures:


[x] Instant dashboard load (IndexedDB)
[x] True offline functionality for course content (student/teacher)
[x] Persistent login (no logout on close)
[x] Resilient media playback (blob offline)
[x] Simple teacher upload (raw MP4 -> Supabase Storage)
[x] No FFmpeg / no transcoding
[x] No custom server / no Cloud Functions
[x] Admin console always shows live data, never stale cache
[x] Realtime updates without manual refresh (row-patched, not refetched)
[x] Institution-agnostic enrollment (manual / prefix / grade-stream)
[x] WiFi-only video caching (saves mobile data)
[x] Resumable chunk downloads
[x] Active storage monitoring (warn/block + persist() quota request)
[x] Graceful degradation under poor connectivity
[x] Offline PDF rendering (pdf.js + blob URLs)
[x] Background sync without user action
[x] Teacher UI shell cached, no content overhead
